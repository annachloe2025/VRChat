    用語集¶

    このプロジェクトで登場する技術用語のまとめ。 新しい用語に出会ったら追記する。

    VRChat / Unity¶

    VRChat SDK3¶

    VRChat のワールドおよびアバターを作るための公式開発キット。 ワールド向けには SDK3 Worlds、アバター向けには SDK3 Avatars がある。 このプロジェクトはワールド制作なので Worlds を使う。

    VCC (VRChat Creator Companion)¶

    VRChat 公式のプロジェクト管理ツール(Windowsアプリ)。

    • 適切な Unity バージョンを指示
    • SDK や追加ツール(UdonSharp、ClientSim など)を一括管理
    • 新規プロジェクト雛形を生成
    • プロジェクトのバックアップ

    2022年末頃に登場し、現在は VRChat 開発の 公式の入り口。 昔の .unitypackage 手動インポート方式は廃止された。

    入手先: https://creators.vrchat.com/

    Udon¶

    VRChat ワールド独自のビジュアルスクリプト言語/ランタイム。 Unity の通常のC#スクリプトはVRChatワールドでは動かないため、Udon を介してロジックを書く。 ノードを線でつなぐビジュアル形式。

    UdonSharp (USharp)¶

    Udon を C# で書くためのコンパイラ。 ビジュアルスクリプトの代わりに、普通のC#コードで Udon プログラムを記述できる。 プログラミング経験者には圧倒的にこちらが楽。VCC からインストール可能。

    ClientSim¶

    Unityエディタ内で VRChat ワールドの動作をテストできるツール。 アバター視点で歩き回れる、UI操作も再現できる。 アップロード前にエディタ上でほぼ動作確認が完結する ようになり、開発サイクルが大幅に高速化した。

    VRCWorld プレハブ¶

    VRChat SDK3 Worlds に同梱されている、ワールドの 基盤となるプレハブ。 スポーン地点、リスポーン処理、基本機能が組み込まれている。 シーンに必ず1つ配置する必要がある。

    Quest 対応¶

    VRChat ワールドを Meta Quest(旧 Oculus Quest)で動くようにビルドすること。 Android(Quest は Android ベース)向けの厳しい制約があり、PC版より:

    • ファイルサイズ上限がきつい(数十MB)
    • 使えるシェーダーが限定的
    • ポリゴン数・テクスチャ枚数の上限がより厳しい

    両対応するには PC 版と Quest 版を別ビルドして、それぞれアップロードする。

    Unity LTS¶

    Long Term Support の略。Unity の 長期サポート版。 通常版より安定していて、サポート期間が長い。 VRChat は LTS 版の特定バージョンを公式推奨にしている(現在は Unity 2022.3 LTS)。

    HDRI¶

    High Dynamic Range Image。輝度差の広いパノラマ画像で、Unity では 空の表現や全体照明 に使う。 リアルな空・光環境を一発で再現できるので、ワールドの空気感作りに重要。

    Git / バージョン管理¶

    Git¶

    ファイルの変更履歴を管理する分散型バージョン管理システム。 コミット単位で「いつ・誰が・何を・なぜ変更したか」を記録できる。

    GitHub¶

    Git リポジトリのクラウドホスティングサービス(Microsoft 傘下)。 バックアップ、共有、公開、プルリクエストによる協業の場として使われる。

    LFS (Git Large File Storage)¶

    Git を 大きなバイナリファイル(画像、3Dモデル、音声等)に対応 させる拡張。 通常の Git は差分管理が前提なのでバイナリで肥大化しがち。 LFS は実体を別サーバに置き、リポジトリにはポインタだけ入れる。

    GitHub の無料枠は アカウント全体で 1GB ストレージ + 月1GB帯域。超えると課金。 このプロジェクトでは LFS は使わず、重いファイルは .gitignore で除外する方針。

    .gitignore¶

    Git に「このファイルは追跡しない」と指示する設定ファイル。 ビルド成果物、作業中の重量バイナリ、キャッシュなどを除外する。

    .gitattributes¶

    Git に「このファイルはこう扱え」と指示する設定ファイル。 改行コード正規化(LF/CRLF)、LFS対象指定、マージ戦略の指定などに使う。

    コミット (commit)¶

    変更内容のスナップショットを履歴に確定する操作。 コミットメッセージで「何をなぜ変えたか」を残す。

    ブランチ (branch)¶

    履歴を分岐させる仕組み。 メインブランチ(main)から派生させて実験し、うまくいったらマージする、という運用が一般的。

    force push (強制プッシュ)¶

    git push -f または --force-with-lease。 履歴を書き換えた後に 無理やりリモートを上書きする 操作。 他人と作業を共有しているリポジトリでは事故の元なので慎重に。

    orphan branch (孤立ブランチ)¶

    過去の履歴を持たない、まったく新しい初期ブランチ。 「履歴を全部消して最新だけ残したい」時に使う。

    shallow clone (浅いクローン)¶

    git clone --depth=1 などで、最新の数コミットだけ を取得する方法。 リポジトリ自体は全履歴を持っているが、ローカルでは省略する。容量と時間の節約に。

    filter-repo / BFG Repo-Cleaner¶

    リポジトリの 過去の履歴を書き換える ためのツール。 誤って入れた巨大ファイルを履歴ごと消したり、機密情報を削除したりするのに使う。

    ドキュメント・サイト構築¶

    Markdown (.md)¶

    軽量マークアップ言語。 記号(#, *, []() など)で見出し・強調・リンクを表現する。 プレーンテキストで書けて、HTML に変換しやすい。

    MkDocs¶

    Markdown ファイルから 静的なドキュメントサイトを生成 するPython製ツール。 mkdocs serve でローカルプレビュー、mkdocs build で公開用HTMLを生成。

    Material for MkDocs¶

    MkDocs 用の テーマパッケージ(別開発者 squidfunk 氏による)。 標準テーマより読みやすく、検索、ナビゲーション、ダークモード等が充実。 このプロジェクトで採用しているテーマ。

    YAML (.yml / .yaml)¶

    設定ファイルによく使われる、人間が読みやすい記述形式。 mkdocs.yml のように、インデントで階層を表す。

    nav (mkdocs.yml の項目)¶

    mkdocs サイトの サイドバーナビゲーションの構造 を定義する設定。 ファイルパスとタイトルを書いて並べると、その通りにメニューが出る。

    admonition (注意書きブロック)¶

    Material for MkDocs の機能で、!!! note "タイトル" のような記法で 目立つ枠付きブロック を作れる。 注意、警告、情報、ヒント、などのスタイルを選べる。

    その他¶

    LTS (Long Term Support)¶

    「長期サポート版」。バグ修正やセキュリティ修正を 長期間提供 するバージョン。 本番運用や安定性重視のプロジェクトで好まれる。

    CI (Continuous Integration)¶

    継続的インテグレーション。コードの変更があるたびに 自動でビルド・テストを走らせる 仕組み。 GitHub Actions や CircleCI などが代表的。 このプロジェクトでは現状未使用。
